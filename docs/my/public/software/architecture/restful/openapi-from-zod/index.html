When choosing a Zod to OpenAPI package, especially with a preference for avoiding language runtimes like JVM (which is the case for JavaScript/TypeScript environments), you'll primarily be looking at TypeScript-based libraries that integrate directly into your Node.js or browser projects.

Here's a comparison of some popular and relevant Zod to OpenAPI packages:

Key Considerations for Comparison

  • Ease of Use/DX (Developer Experience): How intuitive is it to use? Does it extend Zod directly, or require a separate API?
  • Feature Set: Does it support all OpenAPI features you need (paths, components, security schemes, examples, descriptions, etc.)? How well does it handle complex Zod types like unions, intersections, and refinements?
  • Maintenance & Community: Is the library actively maintained? Are there good examples and community support?
  • Flexibility & Customization: Can you easily customize the generated OpenAPI output?
  • Performance: While not usually a major bottleneck for schema generation, it's good to consider for very large APIs.
  • Zod Version Compatibility: Does it support the latest Zod versions (e.g., Zod v3 and v4)?

Popular Zod to OpenAPI Packages

1. zod-openapi (by @samchungy)

  • Overview: This is a widely used and well-regarded library for generating OpenAPI v3.x documentation from Zod schemas. It's often recommended as a modern alternative to zod-to-openapi.
  • Ease of Use/DX: Very good. It extends Zod directly with an .openapi() method, allowing you to inline OpenAPI-specific metadata (description, example, ref, etc.) right within your Zod schema definitions. 
    import { z } from 'zod';
    import { extendZodWithOpenApi } from 'zod-openapi';

    extendZodWithOpenApi(z);

    const userSchema = z.object({
      id: z.string().uuid().openapi({
        description: 'Unique identifier for the user',
        example: 'a1b2c3d4-e5f6-7890-1234-567890abcdef',
      }),
      name: z.string().min(1).max(50).openapi({
        description: 'Full name of the user',
        example: 'John Doe',
      }),
      email: z.string().email().openapi({
        description: 'User\'s email address',
        example: 'john.doe@example.com',
      }),
    });
  • Feature Set: Comprehensive. It supports:
    • Basic types (string, number, boolean, etc.)
    • Objects, arrays
    • Unions (with options for oneOf vs. anyOf)
    • Intersections
    • Enums
    • Literals
    • Dates (with formatting options)
    • Adding descriptions, examples, and component references.
    • Defining full operation objects (request bodies, parameters, responses, headers, callbacks).
  • Maintenance & Community: Actively maintained with a good GitHub presence and a clear API.
  • Flexibility & Customization: Provides options for customizing document creation (e.g., defaultDateSchema, unionOneOf, enforceDiscriminatedUnionComponents). You can register schemas as components and refer to them.
  • Zod Version Compatibility: Designed to work well with Zod v3 and provides guidance for Zod v4 compatibility through dual imports.

2. @asteasolutions/zod-to-openapi (the "older" zod-to-openapi)

  • Overview: This was one of the original and popular choices for Zod to OpenAPI generation. While zod-openapi is newer and often recommended, this library is still maintained and widely used.
  • Ease of Use/DX: Similar to zod-openapi, it also extends Zod with an .openapi() method for inlining metadata. 
    import { z } from 'zod';
    import { OpenApiGeneratorV31 } from '@asteasolutions/zod-to-openapi';

    // Extend Zod (this might be done implicitly or explicitly depending on version)
    // You might need to import 'zod-to-openapi/extend' or call a function.

    const userSchema = z.object({
      id: z.string().uuid().openapi({ example: '...' }),
      name: z.string().openapi({ description: 'Full name' }),
    });
  • Feature Set: Robust. It supports most common Zod types and OpenAPI constructs. It offers generators for both OpenAPI 3.0.x and 3.1.x.
  • Maintenance & Community: Actively maintained.
  • Flexibility & Customization: Allows defining schemas, routes, webhooks, and custom components.
  • Zod Version Compatibility: Provides support for Zod v4 with a beta version.
  • Note: If starting a new project, zod-openapi might be slightly preferred due to its more modern approach and clearer maintenance path as of recent discussions. However, if you're already using @asteasolutions/zod-to-openapi, it's a solid choice.

3. @hono/zod-openapi

  • Overview: This is specifically an extension for the Hono web framework. If you're building your API with Hono and Zod, this library provides seamless integration for generating OpenAPI documentation directly from your Hono routes defined with Zod schemas.
  • Ease of Use/DX: Excellent within the Hono ecosystem. You define your route handlers using Hono's API, and the Zod schemas embedded in those definitions are used to generate the OpenAPI spec. 
    import { Hono } from 'hono';
    import { z } from '@hono/zod-openapi'; // Special import for Hono's integration

    const app = new Hono();

    const userSchema = z.object({
      id: z.string().openapi({ example: '123' }),
      name: z.string().openapi({ example: 'John Doe' }),
    }).openapi('User'); // Register as a component

    app.get(
      '/users/:id',
      z.openapi(
        {
          params: z.object({ id: z.string().min(3) }).openapi({
            param: { name: 'id', in: 'path' },
            example: '1212121',
          }),
          responses: {
            200: {
              description: 'User found',
              content: {
                'application/json': {
                  schema: userSchema,
                },
              },
            },
          },
        },
        async (c) => {
          const { id } = c.req.valid('param');
          // ... logic
          return c.json({ id, name: 'Hono User' });
        }
      )
    );
  • Feature Set: Full OpenAPI generation for routes, including parameters, request bodies, and responses. Leverages the underlying zod-openapi (or a similar logic) for schema conversion.
  • Maintenance & Community: Actively maintained as part of the Hono ecosystem.
  • Flexibility & Customization: Primarily focused on route-level documentation within Hono.
  • Consideration: This is a specific solution for Hono users. If you're using a different framework (Express, Fastify, Koa, etc.), this won't be your primary choice.

4. openapi-zod-client (for generating Zod from OpenAPI)

  • Overview: While the user asked for Zod to OpenAPI, it's worth mentioning openapi-zod-client. This library works in the opposite direction: it generates a TypeScript HTTP client with Zod validation (using zodios) from an existing OpenAPI spec. This is useful if you're consuming an API that already has an OpenAPI spec and you want Zod schemas for client-side validation.
  • No direct Zod to OpenAPI generation.

Recommendation

For directly generating OpenAPI documentation from your Zod schemas in a TypeScript/JavaScript environment (without JVM or other runtimes), zod-openapi (by @samchungy) is generally the most recommended and actively maintained option. It provides a clean API for extending Zod and covers a wide range of OpenAPI features.

If you are using the Hono framework, then @hono/zod-openapi is the natural and most integrated choice.

The "older" @asteasolutions/zod-to-openapi is also a valid choice and has a proven track record, but zod-openapi is often seen as its more modern successor.

Choose the one that best fits your project's existing dependencies and framework choices. All listed options (excluding openapi-zod-client for its different purpose) operate purely within the JavaScript/TypeScript ecosystem, so no JVM or other language runtimes are required.
