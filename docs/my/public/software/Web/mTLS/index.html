mTLS

How mTLS Works

Unlike a standard one-way TLS connection where only the server authenticates itself to the client, mTLS is a two-way authentication process. The key steps are:

  1. Client Hello: The client initiates the TLS handshake by sending a "Client Hello" message.
  2. Server Hello: The server responds with a "Server Hello" and its own server certificate. It also sends a "Certificate Request" message, indicating that it requires a client certificate for authentication.
  3. Client Certificate: The client responds by sending its own client certificate and a "Certificate Verify" message. This message is digitally signed by the client's private key to prove ownership of the certificate.
  4. Server Verification: The server receives the client's certificate and verifies its authenticity using a trusted Certificate Authority (CA). The server checks the certificate's validity period, the digital signature, and whether it was issued by a CA that the server trusts.
  5. Encrypted Tunnel: If the certificate verification is successful, the TLS handshake is complete, and a secure, encrypted tunnel is established. The client and server can now communicate securely.

Setting up a Node.js mTLS Server

To implement this in Node.js, you'll use the built-in tls or https module. You need the following files:

  • Server Private Key (server.key): The server's private key.
  • Server Certificate (server.cert): The server's public certificate, signed by a CA.
  • CA Certificate (ca.cert): The public certificate of the CA that signed the client certificates. The server uses this to verify the client's certificate.
  • Client Private Key (client.key): The client's private key.
  • Client Certificate (client.cert): The client's public certificate, signed by the same CA.

Here is a basic example using the https module:

const https = require('https');
const fs = require('fs');

// Read the necessary certificate and key files
const options = {
  key: fs.readFileSync('server.key'),
  cert: fs.readFileSync('server.crt'),
  ca: fs.readFileSync('ca.crt'), // The CA certificate to verify client certs
  requestCert: true,             // Request a client certificate
  rejectUnauthorized: true,      // Reject connections if the client cert is invalid
};

// Create the HTTPS server
const server = https.createServer(options, (req, res) => {
  console.log("received request");
  
  // Access the authenticated client certificate
  const clientCert = req.socket.getPeerCertificate();

  // If a valid certificate is present, proceed
  if (clientCert) {
    console.log(`Client authenticated: ${clientCert.subject.CN}`);
    res.writeHead(200);
    res.end('Hello from the secure mTLS server!');
  } else {
    console.log(`Client not authenticated: ${req} `);
    // This part of the code is technically unreachable if rejectUnauthorized is true
    // as the connection would have been rejected before the request handler is called.
    res.writeHead(401);
    res.end('Authentication failed.');
  }
});

// Start the server on port 8443
server.listen(8443, () => {
  console.log(`mTLS server listening on https://localhost:8443
you may try cmd:
curl --cert client.crt --key client.key --cacert  ../server/ca.crt https://localhost:8443/abc
curl --cert client.crt --key client.key --cacert  ../server/server.crt https://localhost:8443/abc
`);
});

Step 1: Create a Certificate Authority (CA)

First, you need to create a root CA that will be used to sign both the server and client certificates. This establishes a chain of trust. (in PROD, not same CA?)

  1. Generate the CA Private Key: This command generates a 2048-bit RSA private key for the CA. Keep this file (ca.key) secure and don't share it.

    openssl genrsa -out ca.key 2048

  2. Generate the CA Certificate: This command creates a self-signed CA certificate (ca.crt) using the private key you just generated. The -x509 option indicates it's a self-signed certificate, and -days 365 sets its validity period to one year.

    openssl req -x509 -new -nodes -key ca.key -sha256 -days 365 -out ca.crt

    You'll be prompted to enter information like country, state, and organization. The Common Name (CN) is crucial here; it should be a descriptive name for your CA, for example, My Mutual TLS CA.

Step 2: Create Server Key and Certificate

Next, you'll create a private key and a certificate for your server, which will be signed by the CA you just created.

  1. Generate the Server Private Key: This creates the private key for the server (server.key).

    openssl genrsa -out server.key 2048

  2. Generate a Certificate Signing Request (CSR): A CSR contains information about your server and is used to request a certificate from the CA.

    openssl req -new -key server.key -out server.csr

    Again, you'll be prompted for information. The Common Name (CN) here should be the domain name of your server (e.g., localhost or api.example.com).

  3. Sign the Server Certificate with the CA: This is the most critical step. You're using your CA's private key (ca.key) and certificate (ca.crt) to sign the server's CSR (server.csr), creating the final server certificate (server.crt). The -CAcreateserial option is used to manage serial numbers for the signed certificates.

    openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out server.crt -days 365 -sha256

Step 3: Create Client Key and Certificate

The process for the client is almost identical to the server.

  1. Generate the Client Private Key:

    openssl genrsa -out client.key 2048

  2. Generate a Certificate Signing Request (CSR):

    openssl req -new -key client.key -out client.csr

    For the client, the Common Name (CN) can be anything descriptive, like client.example.com or simply My Client.

  3. Sign the Client Certificate with the CA: Sign the client's CSR (client.csr) using the CA's private key and certificate.

    openssl x509 -req -in client.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out client.crt -days 365 -sha256

    This may also be signed by client-side's own CA, not server side's CA?

After completing these steps, you will have the following files ready for your Node.js mTLS server setup:

  • ca.crt (CA certificate)
  • server.key (Server private key)
  • server.crt (Server certificate)
  • client.key (Client private key)
  • client.crt (Client certificate)

You will use server.key and server.crt on the server side, and ca.crt will be used by the server to verify the client. The client will use client.key and client.crt to authenticate with the server.
