There's no direct built-in command in SQLite FTS5 to "see the actual tokens for a row." FTS5 processes text and stores the tokenized representation internally, but it doesn't expose a simple way to dump the tokens for a specific row.

To debug tokenization, you'll have to reproduce the tokenization process externally or use SQLite's fts5_tokenize function on the same input text.

The fts5_tokenize Method

The fts5_tokenize function is the best way to debug how FTS5 is tokenizing your text. This function is a a virtual table that provides a way to inspect the tokens generated by an FTS5 tokenizer.

Here's how to use it:

-- Step 1: Attach the fts5 extension if not already loaded
.load fts5.dll -- on Windows
.load libsqlite3_fts5 -- on Linux/macOS

-- Step 2: Query the fts5_tokenize virtual table
-- Use the same text you would insert into your FTS5 table
SELECT * FROM fts5_tokenize('unicode61', 'The quick brown fox jumps over the lazy dog.');

Breakdown of the fts5_tokenize query:

  • 'unicode61' is the name of the tokenizer you want to use. This should match the tokenizer used when you created your FTS5 table (e.g., CREATE VIRTUAL TABLE my_table USING fts5(..., tokenize='unicode61');).
  • 'The quick brown fox jumps over the lazy dog.' is the text input you want to inspect.

The result of the query will be a list of tokens, along with their start and end offsets, and their positions within the original text. This allows you to see exactly how FTS5 is splitting your input text into tokens.

Example output:

This method is the most reliable and direct way to debug tokenization issues, such as unexpected tokenization of a hyphenated word or special characters.

Other Debugging Considerations

  • Tokenizer: The behavior of tokenization depends on the tokenizer used. The default is 'unicode61', but custom tokenizers like simple, porter, or icu will produce different results. Make sure to use the correct tokenizer in your fts5_tokenize query.
  • Case Folding: FTS5 performs case folding (converting text to a common case, usually lowercase) as part of tokenization. This is why "The" becomes "the" in the example above.
  • Stemming: If you're using a stemmer (like 'porter'), the tokens will be stemmed words (e.g., "jumping" might become "jump"). This is something else to watch out for.
  • Source Code: For the deepest level of debugging, you'd need to inspect the source code of the FTS5 module itself to understand the tokenization logic. This is an advanced step and usually not necessary for most use cases.

You can see the actual tokens for a specific row by using the fts5() table-valued function. This is the most direct way to inspect how the FTS5 index processed and stored the content for a given record.

## Using the fts5() Table-Valued Function

The fts5() function allows you to query the internal state of an FTS5 index. By providing it with a rowid, you can see the exact tokens stored for that row.

The syntax is: SELECT token FROM fts5('<fts_table_name>', 'tokenize=<tokenizer_spec> <rowid>');

Key Components:

  • <fts_table_name>: The name of your FTS5 virtual table.
  • tokenize=: The command to show tokens.
  • <tokenizer_spec>: The tokenizer used by your FTS table. For the default, this is unicode61. If you've specified a different one (e.g., porter), use that name.
  • <rowid>: The rowid of the row you want to inspect.

### Step-by-Step Example

Let's create a simple table and see it in action.

1. Create the tables and insert data:

-- The original content table
CREATE TABLE articles(title TEXT, body TEXT);

-- The FTS5 virtual table, linked to the content table
CREATE VIRTUAL TABLE articles_fts USING fts5(
  title,
  body,
  content='articles', -- Link to the articles table
  content_rowid='rowid' -- Link using the rowid
);

-- Insert a sample article. Let's say it gets rowid 1.
INSERT INTO articles (title, body)
VALUES ('FTS5 Debugging', 'A quick-brown fox jumps over the lazy dog.');
-- FTS5 tables are automatically populated from the source table.

2. Find the rowid of the document you want to debug:

If you don't know the rowid already, you can find it easily.

SELECT rowid, title FROM articles WHERE title = 'FTS5 Debugging';
-- Returns: 1|FTS5 Debugging

3. Use the fts5() function to see the tokens for rowid 1:

Now, query the virtual table's internal structure for rowid 1. We'll use the default unicode61 tokenizer.

SELECT token FROM fts5('articles_fts', 'tokenize=unicode61 1');

Result:

This query will show you exactly how FTS5 tokenized the content for that row. Notice how punctuation is removed and text is lowercased.

## Alternative: The fts5_tokenize() Auxiliary Function

If you just want to test how a specific piece of text would be tokenized by your index's configuration, without referencing a specific rowid, you can use the fts5_tokenize() function.

This is useful for testing strings before you insert them or for debugging why a specific query term isn't matching.

### Example

Let's test how a different string would be processed by our articles_fts index.

SELECT token FROM fts5_tokenize('articles_fts', 'Testing is 1-2-3.');

Result:

token
testing
is
1
2
3

You can also use this on content from your table, but it's less direct than the first method:

SELECT ftt.token
FROM articles, fts5_tokenize('articles_fts', articles.body) AS ftt
WHERE articles.rowid = 1;
-- This will produce the same list of tokens for the fox/dog sentence as the fts5() example.

## Summary: Which One to Use?

  • Use fts5('<index>', 'tokenize=...'): To see the actual tokens that are stored in the index for a specific existing row (rowid). This is the best method for debugging stored content.
  • Use fts5_tokenize(<index>, <text>): To see how any arbitrary string of text would be tokenized using the index's configuration. This is ideal for "what-if" scenarios and testing query syntax.
