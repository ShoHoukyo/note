sqlite: how to record all update statements received and time each took to run? (better use sqlite built-in)

2025-08-07 by gemini

TLDR: no sqlite sql built-in

Recording every UPDATE statement and its execution time in SQLite can be achieved through a combination of built-in mechanisms. The most robust and programmatic approach involves using a trace callback, while a simpler method exists for interactive sessions via the command-line shell.

Programmatic Approach: Using the Trace Callback

For applications, the recommended method is to register a trace callback function. This function is invoked by the SQLite engine just before the execution of each SQL statement. Within this callback, you can filter for UPDATE statements and measure their execution time.

This functionality is exposed through the sqlite3_trace_v2() C function and corresponding wrappers in various programming languages. Heres how you can implement it in Python using the built-in sqlite3 module:

import sqlite3
import time

def trace_callback(statement):
    # Check if the statement is an UPDATE query
    if statement.strip().upper().startswith('UPDATE'):
        # Record the start time before the statement is executed
        start_time = time.perf_counter()
        
        # We can't execute the statement here, as the callback is called before execution.
        # So, we store the start time and associate it with the statement.
        # A more advanced implementation might use a dictionary to map thread/connection
        # to the last seen UPDATE statement and its start time.
        print(f"Executing UPDATE statement: {statement}")
        
        # This is a conceptual demonstration. To get the end time, you would need
        # to hook into a post-execution callback, which sqlite3_trace_v2 also provides.
        # However, for simplicity, we'll measure the time around the execute call
        # in the main code and correlate it.

def profile_callback(statement, execution_time):
    # This callback receives the statement and its execution time.
    if statement.strip().upper().startswith('UPDATE'):
        print(f"UPDATE Statement: '{statement}' took {execution_time:.6f} seconds")

# --- A more practical implementation using a wrapper ---

class TimedConnection(sqlite3.Connection):
    def execute(self, sql, *args):
        start = time.perf_counter()
        result = super().execute(sql, *args)
        end = time.perf_counter()
        
        if sql.strip().upper().startswith('UPDATE'):
            print(f"UPDATE Statement: '{sql}' took {end - start:.6f} seconds")
            
        return result

# Example Usage
db = sqlite3.connect(':memory:', factory=TimedConnection)
cursor = db.cursor()

cursor.execute("CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT, email TEXT)")
cursor.execute("INSERT INTO users (name, email) VALUES ('Alice', 'alice@example.com')")
cursor.execute("INSERT INTO users (name, email) VALUES ('Bob', 'bob@example.com')")
db.commit()

# Now, let's run some UPDATE statements
cursor.execute("UPDATE users SET email = 'alice_new@example.com' WHERE name = 'Alice'")
cursor.execute("UPDATE users SET email = 'bob_new@example.com' WHERE name = 'Bob'")
db.commit()

db.close()

In the Python example above, we subclass sqlite3.Connection to intercept the execute method. This allows us to measure the time taken for each statement and then filter for and print the details of the UPDATE statements.

For C/C++ applications, you would use sqlite3_trace_v2() with a callback function. The mask argument of this function can be set to SQLITE_TRACE_STMT to be notified of statement execution. To get the execution time, you can also use the SQLITE_TRACE_PROFILE mask, which will invoke the callback after the statement has run and provide the execution time.

Interactive Approach: Using the SQLite Command-Line Shell

For developers working directly with the SQLite command-line interface (CLI), a simpler, interactive method is available using the .timer meta-command.

  1. Start the SQLite CLI:

    sqlite3 your_database_file.db

  2. Enable the timer:

    .timer ON

Once enabled, the shell will display the "real," "user," and "sys" time taken to execute each subsequent SQL statement.

Example Session:

sqlite> CREATE TABLE products (id INTEGER PRIMARY KEY, name TEXT, price REAL);
sqlite> INSERT INTO products (name, price) VALUES ('Laptop', 1200.50);
sqlite> INSERT INTO products (name, price) VALUES ('Mouse', 25.00);
sqlite> .timer ON
sqlite> UPDATE products SET price = 1150.00 WHERE name = 'Laptop';
UPDATE products SET price = 1150.00 WHERE name = 'Laptop';
Run Time: real 0.000 user 0.000088 sys 0.000000
sqlite> SELECT * FROM products;
1|Laptop|1150.0
2|Mouse|25.0
Run Time: real 0.000 user 0.000045 sys 0.000000

While this method is straightforward for manual inspection and debugging, it is not suitable for programmatic logging within an application.

In summary, the most effective built-in way to log all UPDATE statements and their execution times is by leveraging the trace and profile capabilities provided by the SQLite C API, which are accessible through wrappers in most programming languages. For quick, interactive analysis, the .timer command in the SQLite shell is a convenient tool.
