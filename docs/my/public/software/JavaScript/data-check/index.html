zod​

TypeScript-first schema validation with static type inference (how well does it work in other lang? say golang)

p3 check https://github.com/StefanTerdell/zod-to-json-schema https://github.com/StefanTerdell/json-schema-to-zod

vs. io-ts:

  • 2023 very similar functionality, much more readable documentation, more leaning towards OOP rather than FP

https://zod.dev/?id=comparison Zod was designed to mirror TypeScript as closely as possible. But there are many so-called "refinement types" you may wish to check for that can't be represented in TypeScript's type system. For instance: checking that a number is an integer or that a string is a valid email address.

zod had a pretty rich ecosystem where it integrates pretty seamlessly with some libraries e.g. https://github.com/react-hook-form/resolvers#zod

io-ts​

io-ts is very flexible, but also very opinionated towards functional programming, to the point of putting functional programming purity over a fluent developer experience. Its documentation is a bit sparse and lacks details. To use io-ts, you really cannot avoid using fp-ts and functional programming as well. https://javascript.plainenglish.io/a-typescript-runtime-data-validators-comparison-eeedc6b0583a

The advantage of using io-ts to define the runtime type is that we can validate the type at runtime, and we can also extract the corresponding static type, so we don’t have to define it twice. cons: Recursive types can't be inferred by TypeScript so you must provide the static type as a hint decodedPayload._tag === 'Left' weird syntax to check validation-errors Have to make types for DynamoDB https://github.com/gcanti/io-ts/blob/master/index.md
