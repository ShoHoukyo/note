in JS, anything equivalent to python global __debug__? any convention?
2025-09-27 by gemini

JavaScript does not have a direct, built-in equivalent to Python's global __debug__ constant.

However, there is a very strong and widely adopted convention that achieves the same goal: using the process.env.NODE_ENV environment variable.

The Convention: process.env.NODE_ENV

The standard practice in the JavaScript ecosystem (both for backend Node.js and frontend frameworks) is to use an environment variable called NODE_ENV. This variable is typically set to one of two strings:

  • 'development': When you are developing and debugging.
  • 'production': When the code is built and deployed for live users.

You use this variable to conditionally run debug-only code:

// This code will only run during development
if (process.env.NODE_ENV === 'development') {
  console.log('Application is running in development mode. Debug info is available.');
  // Perform detailed validation, log extra data, etc.
}

How It Works: Dead Code Elimination

The real magic happens during the build step, which is what makes this a true parallel to Python's -O flag.

When you build your code for production using a bundler like Vite, Webpack, or Rollup, these tools perform a crucial optimization:

  1. Replacement: The bundler replaces the expression process.env.NODE_ENV with the literal string 'production'.
  2. Evaluation: Your conditional check if (process.env.NODE_ENV === 'development') becomes if ('production' === 'development').
  3. Elimination: The bundler evaluates this to if (false). Since the code inside this block can never run, it is identified as dead code and is completely stripped from the final production bundle.

This means your debugging code doesn't just get skipped at runtime; it is completely removed from the production file, resulting in a smaller, faster application.

Examples

In Node.js

You set the environment variable when you run your script.

# Running in development mode (often the default)
node app.js

# Running in production mode
NODE_ENV=production node app.js

In Frontend Frameworks (React, Vue, etc.)

Frameworks and their build tools (like Vite or Create React App) manage this for you automatically.

  • When you run npm run dev, NODE_ENV is set to 'development'.
  • When you run npm run build, NODE_ENV is set to 'production', and all the debug code is stripped out.

For example, this is why React shows you detailed warnings and error messages in development but not in the production build.

// A common pattern in libraries
function expensiveOperation(data) {
  // In production, this entire 'if' block won't exist in the final code.
  if (process.env.NODE_ENV !== 'production') {
    if (typeof data !== 'object') {
      console.warn('Warning: expensiveOperation expected an object.');
    }
  }

  // ... core logic of the function
}


// write it in .tsx file, rely on build tool
const MyComponent: React.FC<MyComponentProps> = ({ userId }) => {
  // This entire 'if' block will be stripped out of the production build.
  if (process.env.NODE_ENV === 'development') {
    console.log(`Component rendered for userId: ${userId}`);
    // You could also run expensive validation checks here that you
    // don't want to run in production for performance reasons.
  }

how about this globalThis.__debug__?

globalThis.__debug__ can be removed in https://esbuild.github.io/api/#drop-labels but guess process.env.NODE_ENV === 'development' works better

While you can technically create a global variable like globalThis.__debug__, it is not the convention and misses the most important benefit of the process.env.NODE_ENV approach.

The core difference is build-time optimization vs. runtime checking.

The Critical Flaw: No Dead Code Elimination

Using process.env.NODE_ENV allows bundlers (like Vite or Webpack) to perform dead code elimination.

When you build for production, the bundler replaces process.env.NODE_ENV with the literal string 'production'.

  • Your Code: if (process.env.NODE_ENV !== 'production') { console.log('Debug info'); }
  • After Replacement: if ('production' !== 'production') { console.log('Debug info'); }
  • After Evaluation: if (false) { console.log('Debug info'); }

The bundler sees the if (false) block and completely removes it from the final production code. Your debug logic doesn't just get skipped; it ceases to exist in the deployed file, making your application smaller and faster.

A custom global like globalThis.__debug__ cannot be optimized this way. unless instructed in tool like https://esbuild.github.io/api/#drop-labels

  • Your Code: if (globalThis.__debug__) { console.log('Debug info'); }

A bundler has no way of knowing what the value of globalThis.__debug__ will be when the code runs in a user's browser. It could be true or false. Therefore, the if statement and the code inside it must be included in the production bundle. It's a runtime check only.

Other Disadvantages

  1. It's Not a Standard: The entire JavaScript ecosystem, including libraries like React and Vue, relies on process.env.NODE_ENV to enable their own development-only warnings and features. By using a custom flag, you opt out of this ecosystem-wide optimization.

  2. Mutability and Security : A global variable can be changed by any script at any time. A malicious ad script or browser extension could potentially set globalThis.__debug__ = true; on your production site, enabling debug modes you never intended to expose to users. The NODE_ENV check is "baked in" during the build and cannot be changed at runtime.

  3. Manual Setup: You would have to manually write code to set globalThis.__debug__ = true; or false; in each environment, whereas build tools handle NODE_ENV for you automatically.

Summary: Convention vs. Custom Global

Featureprocess.env.NODE_ENV (Convention)globalThis.__debug__ (Custom Global)
OptimizationDead Code Elimination at build timeRuntime check only; code remains in bundle
EcosystemStandard; used by all major tools & libsNon-standard; creates isolated logic
ScopeA build-time constant (immutable in bundle)A mutable global variable
Primary UseCreating optimized production buildsTemporary, interactive debugging in the browser

Conclusion: Stick with the established convention of process.env.NODE_ENV. It's more powerful, safer, and integrated into the entire JavaScript ecosystem. A runtime global flag is only useful for temporary debugging sessions, not for creating distinct development and production builds.
